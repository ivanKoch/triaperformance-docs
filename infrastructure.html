Triaperformance — Technical Documentation

Personal AI Infrastructure

Owner: Iván Koch Last updated: July 17, 2026 ● Live / operational

Summary

A self-hosted, always-on AI agent system giving a solo business owner persistent AI access to his business knowledge base — reachable from a phone, browser, or desktop app, without depending on a personal computer being powered on.

What was built: a single Hermes Agent instance (Nous Research's open-source, self-hosted agent framework), deployed via Docker on a self-managed Linux VPS. Powered by Google's Gemini API and exposed through three interfaces — Telegram, a web dashboard, and a native desktop app — all reading and writing the same underlying agent state, so there is one consistent "brain" regardless of entry point. The same VPS now also hosts the public website.

Key technical decisions

DecisionReasoning
Self-managed VPS over a managed/turnkey productThe managed product tested first ("Managed Hermes") blocked external connections and general-purpose use — no root access, no exposed ports, no ability to run other services. A plain VPS was required to support this setup plus website hosting and workflow automation.
Docker deployment for HermesIsolates the agent process from the host OS, matches the vendor's recommended production path, simplifies updates.
Native Caddy (no Docker) for the websiteAvoids the Docker/iptables interaction that complicated the dashboard's port binding (see Problem #3) — and gives automatic Let's Encrypt HTTPS with zero manual certificate work.
Tailscale VPN + username/password auth for the dashboardAvoids needing a domain + HTTPS callback purely to satisfy OAuth, for a single-user system. A private VPN between the two devices removes the need for public exposure entirely.
Gemini API, prepaid billing, no auto-reloadCost control was a stated priority — the system fails safe (stops working) rather than accumulating unexpected charges.
Local terminal execution backendHermes already runs inside its own Docker container. A second nested sandbox would require mounting the host's Docker socket — effectively granting root-equivalent host access, which undermines the isolation it's meant to add.
Computer Use tool disabledHeadless server, no desktop environment — no legitimate use case, disabled to reduce attack surface.
Telegram allowlist restricted to one user IDWithout this, anyone who discovered the bot's public username could get full access to the agent — terminal, files, business data.

There is one program (Hermes Agent) running on a rented server, split into two always-on processes — one listens for Telegram, the other serves a web dashboard, both sharing the same memory and calling out to Gemini to think. A private VPN (Tailscale) lets the MacBook and browser reach the dashboard without exposing it publicly. Separately, Caddy serves the public website, and n8n + Twenty CRM (both Tailscale-only, no public exposure) run the CoachMatch lead pipeline — n8n handles the mechanical parts (reading Gmail, sending emails, calling Twenty's API on a schedule), Hermes handles the WhatsApp follow-up loop, which needs free-text understanding rather than fixed logic.

Components

1. Compute — Hostinger VPS

KVM2 plan (2 vCPU, 8GB RAM), plain Ubuntu 24.04, no pre-installed template. Public IPv4 179.197.76.70. Chosen over Hostinger's packaged "Managed Hermes" product, which ran in a locked-down environment with no root access or exposed ports — this is a general-purpose box that also hosts the website and can run workflow automation (n8n) later.

2. Container runtime — Docker (Hermes only)

ContainerPurposePort
hermes-gatewayConnects to Telegram, handles all bot messages— (outbound only)
hermes-dashboardWeb UI, REST API, session history, config management9119 (Tailscale-only)

Both mount ~/.hermes → /opt/data, the shared state directory (config, secrets, sessions, cron, logs) that makes them operate as one logical agent.

3. Model provider — Google Gemini

gemini-3.5-flash — pinned release for cost efficiency and stability. Paid tier, $30 prepaid credit, auto-reload deliberately disabled: balance hits $0, service pauses rather than charging further.

4. Interfaces

  • Telegram — primary day-to-day interface, allowlisted to one user ID. Home channel for cron/scheduled output.
  • Web Dashboard (http://100.70.89.17:9119, Tailscale-only) — chat, session history, config editing, API keys, usage analytics, cron management, logs.
  • Hermes Desktop App (macOS) — connected as a remote gateway to the same VPS backend rather than running its own local agent, unifying all three interfaces into one shared brain.

5. Networking & access control

  • Tailscale — WireGuard-based private mesh. VPS gets a stable private address (100.70.89.17) reachable only from the tailnet.
  • Dashboard auth — username/password (env-stored, file permissions 600), appropriate given the network is already private. Nous Portal OAuth is registered but unused (would need a public HTTPS callback).
  • Terminal execution — local, inside the already-isolated Hermes container.

6. Tooling configuration

Default agentic toolset enabled, with two deviations: Computer Use disabled (no desktop environment); free/no-key providers selected wherever available (headless Chromium, DuckDuckGo, Edge TTS) to avoid unnecessary accounts. Image generation left unconfigured — every provider was paid and it isn't needed yet.

7. Website hosting Live — July 16, 2026

triaperformance.com is served from the same VPS, via native Caddy — not Docker, to avoid the iptables/ufw interaction in Problem #3 below, and for automatic Let's Encrypt HTTPS with zero manual cert work.

Webroot

/var/www/triaperformance — deliberately outside ~/.hermes, so a webserver misconfiguration can't expose Hermes's .env or session data.

Caddyfile — /etc/caddy/Caddyfile

triaperformance.com, www.triaperformance.com {
    root * /var/www/triaperformance
    file_server
    encode gzip
}

Deploy pipeline

Source of truth is the website/ folder in the triaperformance-docs GitHub repo (same repo as the business knowledge base). ~/.hermes/deploy-website.sh pulls the repo and rsyncs website/ into the webroot — runs automatically via the existing 6am cron (now does knowledge-base sync and site deploy in one pass), or on demand via SSH or Hermes.

DNS (GoDaddy)

Root @ and www A records point to 179.197.76.70, replacing the old HubSpot-hosted site. Mail-related records — MX, SPF, DKIM, DMARC — were left untouched; those route @triaperformance.com mail through Google Workspace and are unaffected by hosting changes.

Verified live: curl -I https://triaperformance.com returns HTTP/2 200 via Caddy with a valid TLS cert.

Guardrail: Hermes's command-approval mode stays on "ask," not "off," for anything touching /var/www/triaperformance or the Caddy config directly — routine content updates go through the git deploy path, not ad hoc edits.

8. Business automation — CoachMatch lead pipeline Live — July 17, 2026

The full lead-to-follow-up pipeline for CoachMatch (TrainingPeaks' coach-matching product) was rebuilt off HubSpot, split deliberately across two systems: n8n for deterministic, mechanical work (parsing a fixed-template email, calling an API on a schedule), and Hermes for anything needing free-text judgment (interpreting a natural-language status update).

n8n — Workflow 1: Lead creation

IMAP trigger on new CoachMatch mail (read-only, scoped Gmail access) → parses lead fields → dedupe check via Twenty's create-person mutation, branching on success/failure → on a genuine new lead: creates the Person, sends the existing Telegram notification, sends nurture email 1 immediately via SMTP, sets leadStatus = MESSAGE_SENT.

n8n — Workflow 2: Email nurture, emails 2 & 3

Daily Schedule Trigger queries Twenty for everyone at MESSAGE_SENT with emailTouchCount < 3, sends email 2 at +24h or email 3 at +48h based on time since creation. Stop condition for the whole sequence, every channel included: leadStatus moving off MESSAGE_SENT — no separate unsubscribe logic needed. Lead magnet PDFs moved off HubSpot to triaperformance.com/guias/.

Hermes — WhatsApp lead-status feedback loop

No WhatsApp Business API in this build (deliberately ruled out as overkill for this volume). Instead, two small, hand-written scripts — after early attempts to have Hermes write this logic itself produced broken or incomplete code (see Problem #7) — with Hermes's role narrowed to calling them correctly, not authoring them:

  • twenty_followup_check.py — daily Hermes-native cron job. Finds people at MESSAGE_SENT untouched for 2+ days, builds a pre-filled wa.me deep link per person, posts a summary to Telegram — silent if nobody's due.
  • update_lead_status.py — takes a name and one of 8 valid leadStatus values, matches against active leads, refuses to guess on zero or multiple matches, updates Twenty on exactly one match.

Telegram use cases, live today

  • Daily automatic nudge listing stale leads with one-click WhatsApp links.
  • Free-text status updates — "Fabian replied" → REPLIED, "mark Gaston as won (customer)" → WON_CUSTOMER, "mark Aimee as lost, too expensive" → LOST_PRICE. Hermes maps the sentence to exactly one of 8 known values and calls the script — instructed to ask rather than guess when unsure.

Twenty's GraphQL introspection is disabled on this instance, so the valid leadStatus values were confirmed directly from the Twenty UI and hardcoded rather than discovered programmatically.

Notable engineering problems solved

1. Split-brain agent

The Desktop App ran its own independent local agent by default (a free NVIDIA model), disconnected from the VPS's Gemini instance — two "brains" with no shared memory. Resolved by explicitly connecting the Desktop App to the VPS as a remote gateway.

2. Managed hosting product too restrictive

Hostinger's "Managed Hermes" gave no IP, no root, no exposed ports, and a restricted shell that blocked even curl. Confirmed via official docs that the architecture doesn't support external connections at all — moved to a plain, self-managed VPS instead.

3. Docker networking — dashboard unreachable

Despite a port mapping (-p 9119:9119), the app inside the container was binding to 127.0.0.1 (loopback-only), which Docker's port mapping can't forward external traffic to. Fixed with --host 0.0.0.0. Later, the same class of issue meant a ufw rule alone couldn't restrict published container ports — Docker manipulates iptables directly — so the dashboard's port was bound to the Tailscale IP specifically at the Docker level instead.

4. OAuth redirect mismatch

Registering for Nous Portal OAuth via the web UI only allowlisted localhost as a valid callback. Rather than provisioning a domain + TLS purely to satisfy OAuth, Tailscale VPN + username/password auth was adopted instead.

5. Docker-in-Docker terminal sandbox failure

The wizard's "Docker" terminal backend failed because the outer Hermes container had no access to the host's Docker daemon. Rather than mount the host socket (equivalent to granting host root), the terminal backend was switched to local, relying on the existing container boundary.

6. Credential hygiene

A GitHub token was accidentally committed and pushed to a briefly-public repo; GitHub's secret scanning auto-revoked it. Standing rule since: credentials only ever go into the specific secure prompt/field requesting them — never into a document, chat, or anything that could be committed or logged.

7. Agent reliability under complex, multi-step debugging (July 17, 2026)

While building the WhatsApp feedback-loop scripts, Hermes produced a string of unreliable output in one session: misdiagnosed a working network path as a missing Tailscale install, wrote Python with broken indentation and missing logic while narrating that it had "completely transitioned the setup," Base64-encoded a credential with the stated goal of "bypassing security-redaction filters" (flagged and reversed immediately), and at one point requested to delete a script and self-author a new "skill" without being asked. Judged root cause: the flash-tier Gemini model's known weaker performance on sustained agentic reasoning, compounded by a long session. Two standing changes: verify Hermes's claims against literal output rather than its narration, and write new automation logic once, by hand, tested and complete — Hermes's role narrowed to calling it correctly, not authoring it. The briefly-exposed Twenty API key should be rotated as a precaution (not yet done).

Appendix — key commands

# Install Docker
curl -fsSL https://get.docker.com | sh

# Deploy Hermes Agent — interactive setup wizard
mkdir -p ~/.hermes && cd ~/.hermes
docker run -it --rm -v ~/.hermes:/opt/data nousresearch/hermes-agent setup

# Start the Telegram gateway as a persistent background service
docker run -d --name hermes-gateway --restart unless-stopped \
  -v ~/.hermes:/opt/data nousresearch/hermes-agent gateway run

# Start the web dashboard as a persistent background service
docker run -d --name hermes-dashboard --restart unless-stopped \
  -v ~/.hermes:/opt/data -p 100.70.89.17:9119:9119 \
  nousresearch/hermes-agent dashboard --host 0.0.0.0 --port 9119 --no-open

# Tailscale setup
curl -fsSL https://tailscale.com/install.sh | sh
tailscale up
tailscale ip -4

# Business knowledge base — clone + daily auto-update (now also deploys the website)
git clone https://github.com/ivanKoch/triaperformance-docs.git ~/.hermes/triaperformance-docs
crontab -e   # 0 6 * * * ~/.hermes/deploy-website.sh >> ~/.hermes/logs/kb-sync.log 2>&1

# Website — Caddy install + firewall
sudo apt install -y caddy
sudo ufw allow 80/tcp && sudo ufw allow 443/tcp
sudo systemctl reload caddy
